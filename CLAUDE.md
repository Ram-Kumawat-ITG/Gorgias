This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

A full-stack helpdesk supporting multiple Shopify merchants. Support agents manage tickets, view customer/order data, get AI-suggested replies, and track SLA compliance. Multi-channel: email (Mailgun), WhatsApp, Instagram DMs, Twitter/X. External systems (chatbots, third-party apps) can create tickets via an API key–authenticated REST API.

## Commands

### Backend (FastAPI + Python)

```bash
cd backend
python -m venv venv
source venv/bin/activate          # Windows: venv\Scripts\activate
pip install -r requirements.txt
cp .env.example .env              # fill in credentials
uvicorn app.main:app --reload     # dev server on :8000
python -m scripts.create_agent    # initialize first admin user
python -m scripts.register_webhooks  # register Shopify webhooks
python -m scripts.seed_mock_data  # seed test data
```

API docs: `http://localhost:8000/docs` (Swagger UI, auto-generated by FastAPI).

There is no test suite — no `pytest` or similar setup exists in this repo. There is also no linting (no `black`, `ruff`, `flake8`) or Docker setup.

### Frontend (React + Vite)

```bash
cd frontend

cp .env.example .env    # set VITE_API_BASE_URL

```

No ESLint, Prettier, or test runner is configured.

## Architecture

### Backend (`backend/app/`)

- **Entry**: `main.py` — FastAPI app with lifespan hooks (`connect_db` on startup, `close_db` + `close_shopify_client` on shutdown), CORS, 18+ routers. No background jobs run at startup.
- **Config**: `config.py` — Pydantic Settings from `.env`; `MONGODB_URL` and `SECRET_KEY` are required. CORS falls back to `localhost:5173/8000` if `CORS_ORIGINS` is unset — ensure this is set in production.
- **Database**: `database.py` — Motor (async MongoDB); `connect_db()` also creates all 20+ indexes. All DB access is async/await; `_id` fields are MongoDB `ObjectId` and must be converted to string before returning to clients (routers do `t["_id"] = str(t["_id"])` manually).
- **Routers** (`routers/`): One file per domain — `tickets`, `customers`, `orders`, `returns`, `ai`, `macros`, `automations`, `analytics`, `shopify`, `channels`, `email_inbound`, `instagram`, `whatsapp`, `merchants`, `admin_merchants`, `history`, `webhooks`, `media`, `gift_cards`, `external_tickets`, `chatbot_api`, `sla_policies`. Note: `twitter` router file exists on disk but is **not registered** in `main.py`.
- **Services** (`services/`): Business logic — `ai_service.py` (OpenAI/Groq), `llm_client.py` (LLM wrapper), `shopify_client.py` (REST), `shopify_sync.py`, `merchant_shopify.py`, `sla_worker.py` (APScheduler — not currently scheduled), `automation_engine.py`, `mailgun_service.py`, `whatsapp_service.py`, `whatsapp_ai_agent.py`, `instagram_service.py`, `instagram_sales_agent_service.py`, `email_ai_agent.py`, `ai_agent_service.py`, `twitter_service.py`, `api_key_service.py`, `gift_card_service.py`, `retention_service.py`, `activity_service.py`
- **Models** (`models/`): Pydantic v2 document models for MongoDB — no ORM, raw Motor queries
- **Auth**: `routers/auth.py` — authentication is a **shim**: `get_current_agent()` returns the first active agent from DB, or falls back to a hardcoded placeholder `{"id": "local-admin", "email": "admin@local", "role": "admin"}`. No JWT is enforced server-side. External APIs (chatbot, external tickets) use header-based auth instead (see below).
- **Middleware**: `middleware/shopify_hmac.py` — validates Shopify webhook signatures via SHA256 HMAC against `SHOPIFY_API_SECRET`.

### Frontend (`frontend/src/`)

- **Entry**: `main.jsx` → `App.jsx` (route definitions + `ProtectedRoute` checks `localStorage` for token)
- **Layout**: `components/Layout.jsx` wraps all authenticated pages with Navbar + Sidebar
- **Auth**: `context/AuthContext.jsx` + `hooks/useAuth.js`; JWT stored in `localStorage`. Axios interceptor in `api/client.js` auto-attaches `Authorization: Bearer {token}` and redirects to `/login` on 401.
- **HTTP**: `api/client.js` — Axios with base URL from `VITE_API_BASE_URL` env var (not `VITE_API_URL`)
- **Pages** (`pages/`): InboxPage, TicketDetailPage, CustomersPage, OrdersPage, ReturnsPage, AnalyticsPage, settings pages per channel
- **Styling**: Tailwind CSS v3

### Multi-Merchant Architecture

Each merchant has its own record in the `merchants` collection with fields: `shopify_store_domain`, `support_email`, `mailgun_api_key`, `shopify_access_token`, `api_key_hash`, `is_active`, `last_used_at`. Services like `get_whatsapp_config(merchant_id)` and `get_mailgun_config(merchant_id)` look up merchant-specific credentials, silently falling back to global `.env` values if the merchant record is missing or incomplete.

**External API authentication** (chatbot, external tickets): Callers send `X-Shop-Domain` + `X-API-Key` headers. The backend looks up the merchant by domain, verifies the API key against `api_key_hash` (bcrypt), and updates `last_used_at`. Keys are issued via `/admin/merchants/regenerate-key` and shown only once.

### Dual Database

- **Database A** (`MONGODB_URL` / `MONGODB_DB_NAME`): Primary helpdesk DB — all tickets, customers, merchants, etc.
- **Database B** (`MONGODB_B_URL` / `MONGODB_B_NAME`): Optional secondary DB for the external chatbot/AI agent. Connection failures are non-fatal; the app starts with a null DB B handle. Accessed via `get_db_b()`.

### Non-Obvious Patterns and Gotchas

**Placeholder emails for non-email channels**: Tickets from WhatsApp/Instagram/Twitter use synthetic emails (e.g., `{phone}@whatsapp.placeholder`). Code that syncs to Shopify checks `if customer_email.endswith(".placeholder")` to skip the Shopify API call — always preserve this guard when adding new sync logic.

**Shopify sync degrades silently**: `shopify_sync.py` catches Shopify API failures and falls back to cached MongoDB data (or a placeholder customer). Agents see stale data with no UI warning; monitor logs to detect Shopify outages.

**24-hour messaging window**: WhatsApp and Instagram enforce a 24-hour free-form reply window. Services check `is_within_24h_window(ticket.whatsapp_last_customer_msg_at)`. Outside the window, only pre-approved template messages can be sent.

**Per-merchant config with global fallback**: `get_whatsapp_config(merchant_id)` and equivalents look up merchant-specific credentials in DB, falling back to global `.env` values. Missing or deleted merchant configs silently use the global creds — no error is thrown.

**Automation engine stops at first matching rule with `stop_processing: True`**: Rules are evaluated in priority order; when a rule matches and has `stop_processing`, subsequent rules are skipped even if earlier actions in that rule failed.

**SLA worker exists but is not scheduled**: `sla_worker.py` contains `check_sla_breaches()` and the APScheduler setup, but the scheduler was removed from `main.py`'s lifespan. SLA breach detection does not run automatically. The `sla_policies` router (CRUD) is registered; the old `sla` router is not.

**No ORM — raw Motor queries**: All queries use raw dicts. Schema changes require manual migration scripts. ObjectId vs string mismatches are a common source of bugs — routers consistently convert `_id` to string before response.

**AI agents are stateful**: `whatsapp_ai_agent.py`, `instagram_sales_agent_service.py`, and `email_ai_agent.py` maintain conversation state in the database. They are large services (13–75 KB) that orchestrate multi-turn LLM interactions; changes to their prompts or state schema require careful testing.

### Deployment

Both services deploy to Render.com. Config in `backend/render.yaml` and `frontend/render.yaml`. Backend runs `uvicorn app.main:app --host 0.0.0.0 --port $PORT`; frontend is a static site with SPA rewrites (all paths → `/index.html`).

Default admin login (dev): `admin@yourstore.com` / `change-this-password`
